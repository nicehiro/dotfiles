Goal: Replace/augment current Zotero tool so it uses Zotero Web API as primary backend (read/write), with BBT/local .bib as fallback, while keeping a clean interface for search/cite/details/collections.

High-level phases:
1. Clarify requirements and API surface (what commands/tools you want exposed: search, cite, details, collections, auto-tagging, etc.).
2. Design configuration model (env vars for ZOTERO_API_KEY, ZOTERO_USER_ID, optional group IDs; per-machine vs per-project settings).
3. Design the Zotero Web API wrapper (language choice, endpoints, rate limiting, error handling, mapping to current tool actions).
4. Decide integration style with pi (MCP server, skill with helper scripts, or modifying existing zotero tool implementation).
5. Plan incremental rollout and testing (start as separate tool, then switch default, regression tests vs current behavior).

We will adjust/extend this as we go.
